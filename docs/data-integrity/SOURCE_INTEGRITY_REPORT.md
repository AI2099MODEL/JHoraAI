## 1. Scope and Objective
This audit report assesses the mathematical integrity, caching mechanisms, and system interfaces of the JHoraAI platform. Our objective is to verify that the application has exactly one source of truth for every displayed field, separates raw API outputs from derived indicators, enforces secure server-side AI prompt groundings, and uses low-latency local storage persistence layers correctly.

---

## 2. Source-by-Source Integrity Analysis

### SOURCE_A: JHora REST API (The Ultimate Truth)
All raw astronomical data streams (e.g. planet coordinates, divisional chart placements, dasha cycles, and astakoota pairing scores) are generated by the official remote JHora REST engine hosted on Google Cloud Run.
* **Integrity Guard**: `src/lib/jhoraMapper.ts` contains the authoritative mapper `mapJHoraResponseToAstrologyData`. This mapper acts as the gateway validation barrier. If a response does not contain the required nested fields (`divisional_charts`, `graha_dashas`), it immediately aborts the parsing process and throws a descriptive exception: `Invalid JHora response payload`. This prevents corrupted or partial charts from infiltrating the application state.
* **Redundancy Avoidance**: To comply with Phase 9.95 Rule 1, we have removed all secondary, approximate Keplerian calculations (`src/lib/astrology.ts` coordinates) from active user-facing dashboards. The UI depends 100% on SOURCE_A to guarantee sub-arcsecond accuracy, avoiding any discrepancies or math drifts.

### SOURCE_B: Calculated Local Derivations
These parameters are computed by the application's mapper and server endpoints to format coordinate outputs, calculate relative house positions, group planets into graphical house boxes, parse textual lists, and evaluate advanced astrological configurations (e.g. Argalas and Sade Sati stages).
* **Derivation Constraints**: Derived calculations are strictly prohibited from referencing hardcoded constants, raw parameters, or approximate local equations. They are styled as pure functions receiving *only* verified SOURCE_A payloads.
* **No Label Confusions**: The application clearly segregates these calculations in separate columns or sections. For instance, the "Panchanga" card displays both direct API metrics (Tithi, Yoga) and derived items (Gana, Nadi, Varna) with distinctive, clear labeling indicating their derived status.

### SOURCE_C: AI Generated Interpretations
DYNAMIC AI summaries are generated by the server-side **Google Gemini API** (`gemini-3.5-flash` model) using the secure `@google/genai` TypeScript SDK.
* **Grounding & Halo Mitigation**: The Gemini prompt is injected with the high-precision planetary coordinates and active dasha lord sequences. This grounding prevents the AI from fabricating (hallucinating) planetary signs or chart aspects.
* **Visual Boundaries**: All AI responses are rendered in customized conversational cards styled with distinct visual borders, a separate typewriter transition, and an explicit disclaimer footer: *"Interpretive AI Guidance — Grounded in JHora Astronomical Calculations"*.

### SOURCE_D: Hardcoded Placeholders
Static schedules and structural reference parameters (such as the planetary ingress events or specific default Ayanamsa selections) are categorized as SOURCE_D. These values do not change with the input birth details.
* **Visual Representation**: They are styled with an information tip (`Unavailable` or `Static Reference` tags) to denote that they do not represent real-time computations.

## 3. Storage Caching Mechanics
To ensure high performance and an offline-first mobile-friendly experience, we utilize a tiered caching structure:

```
                      +-----------------------------+
                      |      User Form Input        |
                      +--------------+--------------+
                                     |
                                     v
                       +-------------+-------------+
                       |    IndexedDB Check        |  <-- Instant Cache Hit
                       |   (Unique Key Match)      |
                       +-------------+-------------+
                                     |
                       [Cache Miss]  |  [Cache Hit]
                                     v  v
  +----------------------------------+---+    +----------------------------+
  |    HTTPS /api/astrology/calculate    |    |   Instant UI Load          |
  |       (Official Remote Server)       |    |   (No Network Roundtrip)   |
  +----------------------------------+---+    +----------------------------+
                                     |
                                     v
                       +-------------+-------------+
                       |   Write to IndexedDB Cache|
                       |    & localStorage State   |
                       +---------------------------+
```

1. **Client-Side IndexedDB (IndexedDB Store)**:
   * IndexedDB is used to persist historically calculated horoscopes.
   * Every horoscope is indexed by a unique primary key: `date_time_lat_lng_timezone_place`.
   * On submitting birth details, the frontend performs an immediate key query in IndexedDB. If a matching record is found, the system loads the chart instantly without executing a network roundtrip, enabling instantaneous offline browsing.
2. **Current Session Continuity (localStorage)**:
   * The currently active calculated horoscope state is cached in `localStorage` (`jhora_active_astrology_data`).
   * When the user refreshes their browser tab, the application loads the state from `localStorage` immediately, bypassing loading spinners and preserving the exact tab state.
